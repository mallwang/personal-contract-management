# Implementation Plan: Cancellation-Aware Renewals Panel

**Branch**: `010-cancellation-aware-renewals` | **Date**: 2026-06-06 | **Spec**: [spec.md](spec.md)

## Summary

Replace the static 30-day-before-end-date filter in the upcoming renewals panel with a
per-contract calculation that considers each contract's cancellation period. The cancellation
deadline is computed as `endDate − cancellationPeriod` (calendar-accurate, in TypeScript),
and the panel shows a contract starting 30 days before that deadline. The `UpcomingRenewal`
API response shape is extended to expose `cancellationDeadline` (YYYY-MM-DD) and
`daysUntilCancellationDeadline` (signed integer; negative = overdue), replacing `daysRemaining`.
`YEARS` is added to the `CancellationPeriodUnit` enum with a DB migration.

## Technical Context

**Language/Version**: TypeScript 5.x (strict), Node.js LTS (ESM)

**Primary Dependencies**:
- Backend: Fastify, better-sqlite3
- Frontend: React 18, Vite, react-i18next, lucide-react, shadcn/ui (Radix-based)
- Shared: Zod for runtime schema validation

**Storage**: SQLite via better-sqlite3 (file: `data/contracts.db`; tests: `:memory:`)

**Testing**: Vitest for unit and integration tests; Playwright for E2E tests

**Target Platform**: Web application (pnpm monorepo — `packages/shared`, `packages/backend`,
`packages/frontend`)

**Performance Goals**: No specific targets; single-user personal tool

**Constraints**:
- Calendar-accurate month/year subtraction must happen in TypeScript (not SQL) to handle
  edge cases like March 31 − 1 month = February 28/29 (see `research.md`).
- SQLite does not support `ALTER COLUMN`, so the CHECK constraint update requires a
  table rebuild (see `data-model.md` for migration strategy).

**Scale/Scope**: Dozens to hundreds of contracts; no scale concerns.

### Step 3: Backend — DB Migration for YEARS

**File**: `packages/backend/src/db/client.ts`

Add a new migration step after the `hasAnonymize` block:

1. Query `sqlite_master` for the `contracts` table SQL.
2. If `'YEARS'` is not in the constraint string, rebuild the table:
   - `CREATE TABLE contracts_new` with updated CHECK
     `IN ('DAYS','WEEKS','MONTHS','YEARS')`
   - `INSERT INTO contracts_new SELECT * FROM contracts`
   - `DROP TABLE contracts`
   - `ALTER TABLE contracts_new RENAME TO contracts`

**Write tests first** (`packages/backend/tests/unit/migration.test.ts`):
- Test that a DB migrated from an old schema accepts a contract with `YEARS` as
  `cancellation_period_unit`.
- Test that the migration is idempotent (running it twice does not error).

### Step 4: Backend — Rewrite `getUpcomingRenewals` (TDD)

**File**: `packages/backend/src/services/dashboard.ts`

**Write failing tests first** in `packages/backend/tests/unit/dashboard.service.test.ts`,
replacing the existing `DashboardService – upcomingRenewals` suite:

Key test cases:
- Contract with 3-month cancellation, end date 4 months out → **appears**
- Contract with 3-month cancellation, end date 6 months out → **absent**
- Contract with no cancellation period, end date 20 days out → **appears** (default window)
- Contract with no cancellation period, end date 40 days out → **absent**
- Contract whose cancellation deadline is already past (overdue) → **appears** with negative
  `daysUntilCancellationDeadline`
- Contract with 14-day cancellation, end date 44 days out → **appears** (entry = 44−14−30 = 0 days)
- Contract with 1-year cancellation, end date 13 months out → **appears**
- Contract with YEARS unit works correctly
- Contracts sorted by `daysUntilCancellationDeadline` ASC, name ASC as tiebreaker
- LIFETIME contracts excluded
- Contracts whose `endDate` has already passed excluded (non-overlap with expiredContracts)
- Empty panel returns empty array

**Then implement**:

```
getUpcomingRenewals():
  1. SELECT all contracts WHERE end_date IS NOT NULL
        AND billing_interval != 'LIFETIME'
        AND end_date >= DATE('now')
     Include: id, name, category, end_date,
              cancellation_period_value, cancellation_period_unit
  
  2. For each row:
     a. Parse endDate
     b. Compute cancellationDeadline:
          if cancellationPeriod is null → cancellationDeadline = endDate
          else → subtract period from endDate (calendar-accurate)
     c. Compute panelEntryDate = cancellationDeadline − 30 days
     d. Filter: today >= panelEntryDate (i.e., contract is now visible)
     e. Compute daysUntilCancellationDeadline = round((cancellationDeadline − today) / 86400000)
  
  3. Sort ascending by daysUntilCancellationDeadline, then name
  
  4. Return array of UpcomingRenewal items
```

Extract a pure `computeCancellationDeadline(endDate, period)` helper for the date arithmetic
(unit-testable without DB).
